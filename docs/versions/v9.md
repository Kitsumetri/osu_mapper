## Round 1 (researched in parallel)

### 9.1 Postprocess 1/4-snap — a real BPM-dependent bug, FIXED (`bc3c80a`)
Detail: [v9/task1_postprocess.md](v9/task1_postprocess.md). The user's "I have to nudge notes onto 1/4
in the editor" was a genuine defect. `snap_to_grid` stored `o.time + int(round(grid − o.time))`; on a
half-tie delta, banker's rounding (`round(±0.5)→0`) left the note 1 ms off the editor's own integer
tick → editor flags it **unsnapped**. Measured **1.8%–52.9% unsnapped depending on BPM/offset** (worst
205 BPM); the fix (store `int(round(grid))` directly) drops it to **0%** on every sample. + all-objects
playfield guard (`clamp_objects_to_playfield`, circles too) + NaN-safe `_clamp` + 8 hermetic tests.
- **Ruled OUT (not bugs):** the bounded snap (`max_snap ≈ beat/16`) is correct (0% of objects off all
  of {1/4,1/8,1/6} by >1 ms; it protects against a wrong BPM estimate); circle OOB via postprocess
  can't happen.
- **Open → v9 model/decode (NOT postprocess):** ~16% of gaps on a *straight* song land on the 1/6
  (triplet) grid — the model's onset noise caught by the 1/6 divisor, not a snap bug. Did **not**
  force-snap to 1/4 (would wreck real triplets/streams). Candidate fixes: per-song divisor selection or
  tighter onset precision.

### 9.3 "Ranked-map" reward + RL alignment — design (`c09fabf`, `src/eval/reward.py`)
Detail: [v9/task3_rl_alignment.md](v9/task3_rl_alignment.md). A computable reward `R(map, target_SR)`.
- **Reward (one line):** `R = 0.65·(weighted-mean band-membership inside the real per-SR p10–p90 bands)
  + 0.35·(rosu SR-closeness)`. The per-metric sub-score is a **flat-topped tent** — `1.0` *anywhere
  inside* the band, falloff only outside. This is the anti-reward-hacking core: **zero gradient for
  going more extreme**, so the optimum is "land in the ranked distribution", not "maximise jumps".
  Encodes the v8 `--spacing-scale` lesson (maximising a spacing metric played *worse*).
- **RL ladder (cheapest-first):** best-of-N (no train) → RWR / best-of-N distillation → Diffusion-DPO →
  DDPO/DPOK (only if needed). DRaFT/reward-guidance blocked (non-differentiable reward).
- **The strategic call — conditioning FIRST, then RL** (synergistic). Both attack mean-regression from
  opposite ends; §10.11's diagnosis says the missing lever is *new information at the input* = per-song
  conditioning. RL alone pushes spacing **globally** (the `--spacing-scale` failure). Conditioning makes
  "more jumps *on the jumpy song*" expressible; RL then commits to that tail.

### General (pattern-balanced) reward (`61535cd`)
Detail: [v9/task_general_reward.md](v9/task_general_reward.md). The old `quality` was a flat mean over
14 metrics → the **spacing/aim family was 31% of quality** and out-voted sliders (the user's sr5 bon

over **5 pattern families** (rhythm / spacing_aim / flow / slider_shape ≈ 22.7% each, accents 9.1%),

Flat-topped bands preserved metric→family→quality (strictly harder to game). **Gold calibration (K=500
ranked maps): reward mean 0.953 / median 0.976**, balanced across SR buckets and families = the
"ranked maps of any style score high" property. API unchanged (`RewardBreakdown` gains
`family_breakdown`). Calibration tool: `src/eval/measure_reward.py`.

### RL / policy-gradient feasibility (`5be40f7`, prototype `src/rl/sample_logprob.py`)
Detail: [v9/task_rl_policy_gradient.md](v9/task_rl_policy_gradient.md). **Verdict: DDPO/DPOK are
feasible on 12 GB but NOT the first move.** Three compounding reasons: (1) the production `ddim_sample`
is `eta=0`/deterministic → **no per-step log-prob to differentiate**; (2) the **flat-topped band reward
starves a policy gradient** — advantages collapse to zero in-band, so PG can only pull outliers inward
(the reward is built for *selection*/*reweighting*, not gradient-climbing); (3) even the fit-on-12 GB
config (LoRA + detached rollout + per-step recompute + 10-step eta=1 + grad-ckpt) is ~days of GPU and
RL-without-conditioning pushes spacing *globally*. **→ Phase-1 RL = RWR / best-of-N distillation on the
*conditioned* model.** The prototype proves the v-pred+zero-SNR+CFG per-step log-prob math is correct
and gradients flow (model files untouched; exact `diffusion.py` additions spec'd in the doc).
